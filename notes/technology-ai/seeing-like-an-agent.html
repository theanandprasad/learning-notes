Seeing Like an Agent (Lessons from Building Claude Code)

By Thariq (@trq212, Anthropic) • Published Feb 28, 2026 • Saved Jun 27, 2026
Claude Code Tool Design Progressive Disclosure Anthropic

One of the hardest parts of building an agent isn't the model — it's the action space you hand it. Give Claude one tool or fifty? The right answer depends on abilities you can only learn by watching the model work.

The crux

Claude acts on the world through tool calling, built from primitives like bash, skills, and code execution. The central design question is the shape of its action space: do you give it one general tool (write code / run bash) or fifty narrow tools (one per use case)? The answer is to design tools shaped to the model's own abilities — and the only way to know those abilities is to pay attention: read its outputs, run experiments, watch where it stumbles. Over time, "you learn to see like an agent."

The math-problem framework

Imagine you're handed a hard math problem and asked what tools you'd want to solve it. The honest answer is: it depends on your skills. The right tool isn't the most powerful tool that exists — it's the most powerful tool you can actually wield. That same logic governs how you equip an agent.

A weaker solver handed a computer flounders; a strong solver handed only paper is throttled. Tool design is the act of matching the rung of this ladder to what the model can do today — and revisiting it as the model improves.

Lesson 1 — Improving elicitation: the AskUserQuestion tool

The goal was to improve Claude's ability to ask questions — elicitation. Claude could already ask in plain text, but answering felt slow. The team wanted to lower the friction and raise the bandwidth between the user and Claude.

Attempt #1 — bolt it onto ExitPlanTool. Add a parameter holding an array of questions alongside the plan. Easiest to implement — but it confused the model, which mixed up planning with asking.
The working version — a dedicated tool. A structured-output tool that makes Claude present the user with multiple options, and is composable: callable from the Agent SDK and referenceable inside skills.
"Most importantly, Claude seemed to like calling this tool… Even the best designed tool doesn't work if Claude doesn't understand how to call it."

Is this the final form of the tool? Unsure. What works for one model may not work for another — the design is contingent on the model in front of you, not a permanent truth.

Lesson 2 — Updating with capabilities: Tasks & Todos

At launch, the model needed a scaffold to stay on track over long tasks. So the team gave it TodoWrite: a tool to write and update todos, display them to the user, and check them off as work progressed.

But Claude still sometimes forgot what it was supposed to do. The fix wasn't another tool — it was system reminders injected to keep the model oriented. The broader point: you adapt the harness as the model's capabilities change. A scaffold that's load-bearing today may be redundant tomorrow, and a gap that appears today may need a reminder, not a rebuild.

Lesson 3 — Progressive disclosure: building context

The team gave Claude search capabilities — instructions on how to use an API, how to query a database. Over the course of a year, Claude went from barely being able to assemble its own context to doing nested search across several layers of files to locate exactly the context it needs.

"Progressive disclosure is now a common technique we use to add new functionality without adding a tool."

This is the key unlock. Instead of widening the action space every time you want new behavior, you let the model discover what it needs, on demand, by searching deeper. New functionality, no new tool.

Lesson 4 — The Claude Code Guide Agent (disclosure in action)

Claude Code has roughly 20 tools, and the bar to add a 21st is deliberately high — "this gives the model one more option to think about." Every tool added is cognitive overhead spent on every turn.

A concrete problem: Claude didn't know enough about how to use Claude Code itself — how to add an MCP server, what a slash command does. One option was to stuff all of that into the system prompt. But since users rarely ask, that knowledge would sit in context unused — adding context rot and interfering with normal operation. So instead the knowledge is surfaced on demand via a guide agent (progressive disclosure) rather than bloating the always-on context.

Add another tool / stuff the system prompt

  • Knowledge sits in context whether or not it's needed.
  • Context rot — rarely-used info dilutes attention.
  • More tools = more options to weigh every single turn.
  • Interferes with normal operation for the common case.

Progressive disclosure / model state via tools

  • Lean, always-on context — only the essentials persist.
  • Specialized knowledge loaded on demand when relevant.
  • Action space stays small; the guide agent absorbs the niche.
  • New capability without a new top-level tool.

The pattern generalizes: when you're tempted to add a tool or pad the prompt, ask whether the model can instead retrieve that capability the moment it needs it. Keep the standing context lean; let depth be opt-in.

The throughline

Every lesson is the same move applied differently: watch what the model can do, then shape the action space to fit. Give a behavior its own tool when it deserves one (AskUserQuestion). Scaffold what the model can't yet hold, and adapt as it grows (TodoWrite + reminders). And whenever possible, add capability through disclosure rather than surface area (nested search, the guide agent). That habit of attention is what it means to see like an agent.
