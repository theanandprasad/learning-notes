Hermes Harness Architecture

By Aparna Dhinakaran — Co-founder & CPO, Arize AI • Published May 29, 2026 • Saved May 31, 2026
AI Agents Open Source Hermes / Nous Research Harness Architecture

A teardown of Hermes (Nous Research) — "one of the best open-source harnesses in the ecosystem right now" — mapped against the nine-part harness model, plus the three systems that sit outside that frame.

The thesis

Hermes implements all nine components of the harness model from the earlier "What is an Agent Harness" piece. The interesting part is how it implements them — and the extra subsystems (messaging gateway, profiles, cron) that push it toward long-lived agent workloads most coding harnesses can't handle.

Builds on What is an Agent Harness (the framing) and is informed by Swarm management in agent harnesses plus Anthropic's posts on Managed Agents.

1 · The outer iteration loop

The core loop is familiar: model call → tool dispatch → append tool result → repeat until a final response or interrupt. Where Hermes is stronger than most open harnesses is provider abstraction.

The same runtime can drive chat-completions APIs, Anthropic Messages, Codex Responses, an out-of-process Codex app-server path, and Bedrock. Tool-call formats and provider quirks are normalized by transport adapters, so at the loop level the model surface looks consistent.

2 · Context management & compression

Not a simplistic context trim — a full compression path:

  • Older turns are summarized by an auxiliary model.
  • Head and tail segments are protected by token budget.
  • Tool outputs older than a configurable threshold are pruned before summarization.
  • Summary budget scales with compressed content at ~20%, with a 2,000-token floor and a 12,000-token ceiling — avoiding wasted context on tiny compressions while giving big ones enough room.

The standout idea: compression is a session lifecycle event. On compaction, Hermes closes the current SQLite session row, creates a child session seeded by the summary, rotates the session ID, and records parent-child lineage. A long conversation that compresses repeatedly becomes a lineage chain, not one repeatedly-rewritten transcript. The author calls this "pretty unique" among reviewed harnesses.

3 · Skills & tools management

Hermes separates two concerns most harnesses conflate: tool registration vs. tool exposure.

4 · Subagent management

Delegation primitives are solid: a child run gets its own task ID, its own terminal context, and returns a structured summary to the parent. Dangerous commands default to deny in delegated contexts, and recursion depth is capped.

The current limit: lifecycle ownership

Most child work still lives under the parent call path — when the parent is done, the child is done. There is not yet a durable, externally-steerable child-run plane with independent control semantics. (See "Where Hermes should go next.")

5 · Session persistence & recovery

This is where Hermes departs most from editor-first harnesses. Session state lives in SQLite with FTS5 search and WAL journaling (with fallback for filesystems that can't support WAL). Sessions track source tags for turns, parent-child lineage for compression splits, and metadata the gateway uses to resolve routing before the model runs.

The effect: sessions become runtime infrastructure, not just transcripts for resume.

6 · System prompt assembly

Hermes explicitly composes the system prompt in three tiers. The tiering is explicit in code, which makes invariants easy to reason about and keeps prompt prefixes cache-friendly across normal turns (rebuild is tied to compression / invalidation points).

7 · Lifecycle hooks

Two hook surfaces with different trust models, both serving the same purpose: policy enforcement, auditing, and host side-effects that execute independently of model cooperation.

8 · Permission & safety layer

Woven throughout: dangerous commands default to deny in delegated contexts, the same permissions machinery gates interactive sessions and cron jobs, and the context tier runs prompt-injection scanning on project files before they ever load. Permissions are treated as a cross-cutting plane, not a per-tool afterthought.

Where Hermes should go next

The obvious next step is moving from strong delegation to first-class orchestration.

delegate_task already produces useful worker behavior and structured returns. What's still missing is durable child-run control:

  • run IDs
  • explicit lifecycle management
  • external steering
  • cleanup semantics that survive parent completion

Hermes already has much of the substrate (session infrastructure + gateway routing). Promoting child runs to first-class control-plane objects is the natural next step — and a gap versus the OpenClaw orchestration layer.

To close

Hermes already has excellent execution quality and a strong systems foundation for an open harness — "probably one of the best open model harnesses in the ecosystem." The pace of shipping is "insane," and most of the hard substrate work is already in place. It can already support much broader long-lived agent workloads than typical coding harnesses handle today.
