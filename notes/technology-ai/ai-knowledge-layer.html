The AI Knowledge Layer

By Shann (@shannholmberg) · Published Apr 14, 2026 · Saved Jun 6, 2026
AI Agents Knowledge Management Context Engineering Open Source

Why your agents are useless without it — a two-layer system that makes every agent you run smarter. 20-minute setup, compounds daily, fully open source.

The core idea

The knowledge layer is infrastructure that sits between you and your agents — what they read before they do anything. Two parts: the Knowledge Base Layer (KBL), dynamic and agent-maintained, and the Brand Foundation (BF), static and human-edited. Without it, agents guess. With it, they know.

What it is — two layers

Every conversation with an agent normally starts from zero. You re-explain your business, your voice, your goals, your context — and the output is generic because the input has no memory. The knowledge layer fixes that by giving agents something to read first.

Knowledge Base Layer (KBL) — dynamic. Dump raw sources into a folder: tweets, articles, bookmarks, PDFs, notes, voice memos. An agent reads all of it, classifies each source by type, builds structured wiki pages with cross-references, and maintains a master index of one-line TLDRs for fast scanning. Every question you ask gets filed back as a new page, so the wiki gets richer over time.

Brand Foundation (BF) — static. The layer only you edit: voice rules, visual style, positioning, audience definition, banned words. Agents read it before producing anything but never rewrite it. It's the anchor that keeps everything sounding like you.

Shann tested approaches over two weeks, simplified to the one that works, and loaded his own data — notes, ideas, tweets, articles, bookmarks. The agent compiled 230+ structured wiki pages, cross-linked with concepts, entities, and sources he can query anytime. Karpathy had recently described shifting most of his token spend from code to knowledge management.

Why not RAG?

RAG re-derives answers at query time by chunking documents and searching. The knowledge layer instead compiles once, cross-references, and stays current. At roughly 100 articles, Karpathy found the compiled approach outperforms RAG for Q&A. The graphify project measured 71.5× fewer tokens per query versus searching raw files.

Why most people won't build one

Same reason most people won't meal prep on Sunday: one hour upfront saves ten over the week, but the friction lands today and the payoff lands later. About 90% of people using AI just prompt, accept, and ship — no judgment, no taste. The layer is how you leave that 90%: it encodes your taste, data, and patterns, so agents produce output that sounds like you wrote it instead of AI slop.

The catch is curation. Garbage in = garbage out. If a source isn't 80%+ relevant, skip it. High-signal inputs make better output; noise in is noise out.

The folder structure

Everything lives in a single vault. raw/ is the messy inbox; wiki/ is the compiled output; CLAUDE.md is the schema that controls everything.

my-wiki/ ├── raw/ # messy inbox — drop anything here │ ├── clippings/ # ← Obsidian Web Clipper drops here │ ├── ideas/ │ ├── bookmarks/ │ ├── articles/ │ ├── papers/ │ ├── x-archive/ │ └── assets/images/ ├── wiki/ # compiled, cross-referenced pages │ ├── index.md # master index — one-line TLDRs │ ├── log.md # append-only changelog │ ├── concepts/ │ ├── entities/ │ ├── sources/ # one summary per raw source │ ├── outputs/ # filed answers │ ├── sops/ │ ├── syntheses/ │ └── templates/ └── CLAUDE.md # the schema that controls everything

The Obsidian Web Clipper drops pages into raw/clippings/ with the source URL in frontmatter. Then /wiki-ingest reads the URL, detects the type (tweet / article / PDF / video), sorts it to the right subfolder, fetches the full content, downloads images, and compiles it into the wiki. Clip during the day, ingest overnight, wake to a richer base.

How agents plug in

  • Writer agent — reads the BF for voice, queries the wiki for research, checks performance data to pick the format.
  • Researcher agent — monitors X / Reddit / YouTube, feeds raw/, expands topics.
  • Content strategist — cross-references what's performing against existing coverage to find content gaps.

"Each agent is only as good as the knowledge layer it reads from." Same agent + a thin KB = slop. Same agent + 200 structured pages carrying your voice, data, and history = work that sounds like you. (Shann tested Helena, an AI marketer that profiles a brand from a URL — the brand read was accurate, but content alone was "a whole month of marketing slop in a minute." You need the knowledge layer on top.)

The 5 levels of AI

Scales from personal → team → org

The architecture is identical at every scale; only the number of people reading and writing the layer changes.

Personal
one person, one wiki
→
Team
5–10 people, shared base
→
Org
50+, role-tuned agents on a "central brain"

Raw in → agents compile → cross-refs build → humans validate via exploration gates (every page starts unreviewed; only a human marks it confirmed). Git version-controls everything — synced, reversible. New members point their agent at the base and are productive day one.

The name varies by who's describing it: Karpathy calls it an "LLM wiki"; Eric Osiu calls it a "shared brain" ("a personal assistant has a ceiling; the unlock is role-tuned agents sharing context through a central brain"); Cody Schneider built a 10-capability GTM agent swarm on a "data warehouse" ("without it, agents can't make good decisions"). Greg Isenberg's "ambient businesses" run primarily on agents, with Medvi cited as proof ($1.8B revenue run rate, 2 employees). The label doesn't matter — agents need compiled, structured knowledge.

The 20-minute setup

1 · Clone the repo (2 min). git clone github.com/shannhk/llm-wikid my-wiki, then open it as an Obsidian vault.
2 · Run your agent. Point Claude Code (or any agent that reads markdown + runs bash) at the folder. It reads CLAUDE.md — the schema — and scaffolds the wiki structure. Key operations: /wiki-ingest (process raw into pages), /wiki-query (ask, get cited answers), /wiki-explore (browse + validate), /wiki-lint (catch contradictions, stale content).
3 · Fill it up (10 min). Request your X archive, grab the Obsidian Web Clipper, brain-dump for 10 minutes, dump your bookmarks. Anything 80%+ relevant only goes into raw/.
4 · Run /wiki-ingest (5 min). It sorts clippings, fetches full content from URLs, downloads & analyzes images, classifies by type, builds pages with cross-refs plus counter-arguments and data gaps on every page, and updates the master index.
5 · Open Obsidian graph view and start querying with /wiki-query.

Example query — /wiki-query "what content formats get the most bookmarks?" reads the index TLDRs plus relevant pages and returns a cited answer ("Threads with numbered lists get 3.2× more bookmarks…") with sources, filed back as a new page.

Quality controls

  • Bias checks — force counter-arguments and data gaps on every page; if 10 articles all agree, it flags what's missing.
  • Validation gate — every AI-built page starts explored:false until a human reviews it.
  • Confidence levels — high / medium / low / uncertain.

The 80/20 rule: let AI do 80% of the organization, compilation, and cross-referencing; apply your taste to the last 20% — curation, validation, and the connections only you can see.

What to do next: schedule a trigger (Claude Dispatch or cron) to run /wiki-ingest each morning on yesterday's clips; run /wiki-lint every week or two; at 300+ pages install qmd (by Tobi Lutke) for local hybrid search (BM25 + vector + LLM re-rank, with an MCP server); then connect the layer to your writer / researcher / strategist agents.

The window

The demand is obvious: Karpathy's post hit 99,000+ bookmarks; graphify shipped in 48 hours and got 27,000+ more; multiple implementations went viral the same week. Most people will bookmark and never build one. The ones who spend 20 minutes today get a compounding base by next month that knows their voice, data, and taste — something no search engine or generic prompt can replicate.
