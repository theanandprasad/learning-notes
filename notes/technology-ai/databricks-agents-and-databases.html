Databricks on Agents & Databases

Matei Zaharia & a Databricks co-founder · Latent Space podcast (swyx) • Published 2026 · Data + AI Summit • Saved Jul 7, 2026
Agents Databases Data Infrastructure Databricks

A wide-ranging conversation with two Databricks co-founders that keeps returning to one idea: the models are good enough now — the bottleneck is getting the right data into the right place so agents can act on it, safely and cheaply.

The thesis of the whole episode

"Once you can get the data in the right place, the AI models are becoming pretty good… a lot of traditional software will be rewritten with this new paradigm — get the data there, slap some AGI on top, magic comes out. But without the right data, you can't really do that." Everything below serves that: make data available in the right format, then let agents operate on it — portably, controllably, cheaply.

New here? A plain-English decoder

This conversation moves fast through a lot of jargon. Here's what the building-block terms mean and why anyone uses them — skim it, then the rest reads easily.

The meta-harness (Omnigent)

Concept — agents, harnesses, and a "meta-harness"

A plain chatbot is one-shot: you ask, it answers. An agent instead runs in a loop — it reads context, calls a tool (read a file, run code, query an API), sees the result, and decides the next step, repeating until the task is done. The harness is the scaffolding around the model that makes that loop work: it defines the tools, executes the model's requested tool calls and feeds results back, and manages context and memory. The model decides; the harness does — Claude Code, Codex, and Cursor are harnesses driving a model. A meta-harness, then, is a layer above those harnesses so you can run and switch between them through one interface.

Teams had built 5–6 near-identical internal agent frameworks, and everyone kept having to swap model + harness "every few months." An agent is also "useless if you can't share sessions, keep history, search, and collaborate" on top of it. Omnigent is the answer: an open-source meta-harness — a layer above individual coding agents (Claude Code, Codex, Cursor CLI, Antigravity) that makes them portable, secure, and collaborative.

Securing agents: contextual (stateful) policies

Concept — prompt injection (the threat this defends against)

Prompt injection is the defining security flaw of LLM apps. A model has no reliable way to tell its instructions apart from the data it's reading — to it, both are just text in the same window. So attacker-controlled text hidden in that data can hijack it. Direct injection is a user typing "ignore your rules" (jailbreaking); the dangerous kind for agents is indirect injection, where the poison sits in content the agent fetches on its own — a web page, an email, a file, a tool result — and quietly tells it to, say, grab a secret and send it to an attacker's server. It's ranked the #1 risk in the OWASP LLM Top 10 precisely because a tool-using agent can be talked into acting on the injected instruction (exfiltrating data, running destructive commands) within its own permissions. A recommended defense is to check each proposed action against the user's original intent — which is exactly what stateful policies enable.

The security centerpiece. Most coding agents today allow/deny per tool, statically — which puts you in a bad spot, because individually-safe actions become dangerous in combination.

Because policies are stateful, one tracked state is $ spent per session. An agent once burned $500 reading log files; now you can say "launch a sub-agent, cap it at $5, ask me for more" and it pops up at the cap. Security ranks above cost for a cloud provider, but "token-maxing" matters when 100k engineers each add $1k/month.

The database split: OLTP, OLAP, and the pain of CDC

"You cannot vibe-code with just markdown files" — once an app is serious, it needs a database. But no single database is great at everything, so the field split long ago into two camps tuned for opposite jobs.

Concept — OLTP vs OLAP, and why data is stored two ways

OLTP (Online Transaction Processing) is the live database behind an app. It handles a flood of tiny, concurrent operations that each touch a few rows — place an order, update a balance — and is tuned for millisecond reads/writes and strong consistency so you never lose or corrupt data. Postgres, MySQL, and Oracle are OLTP databases; every serious app has one.

OLAP (Online Analytical Processing, aka "analytics") exists because those same systems are terrible at business questions like "revenue by store, per month, over three years." Such a query scans millions of rows and does heavy aggregation; run on the live database it would both crawl and starve real customer traffic. So analytics gets its own system, tuned for a few big read-heavy scans instead of many small writes.

Why they can't be one system — physical layout. A row store (OLTP) keeps each record's fields together on disk, so grabbing or updating one whole row is a single local read — ideal for transactions. A column store (OLAP) keeps every value of a column together, so a query that sums two columns across millions of rows reads only those two columns instead of dragging every full row through memory — far less I/O, far faster scans. Columnar also compresses roughly 5–10× better (a column holds one data type with lots of repetition, which encoders like dictionary and run-length compression love), which is why analytics data is stored column-oriented. The trade-off is symmetric: columnar makes single-row writes expensive, so row stores back OLTP and column stores back analytics.

Because you now have two separate databases, you have to keep the analytics copy in sync with the live one — ideally continuously, not by re-dumping the whole table every night. That job is the source of the episode's biggest groan.

Concept — CDC (Change Data Capture) and the write-ahead log

Change Data Capture is the standard way to stream a source database's row-level changes (inserts, updates, deletes) into downstream systems — a warehouse, a search index, a cache — incrementally, so they stay fresh without expensive full reloads. The clever trick it relies on: every database already keeps a change journal.

That journal is the write-ahead log — Postgres calls it the WAL, MySQL the binlog. The database writes each change into this append-only log before applying it to the actual data files (hence "write-ahead"), so that if it crashes mid-write it can replay the log on restart and lose nothing. It exists for durability and crash recovery — and it's also what powers replication (standby copies just re-apply the log).

Log-based CDC (e.g. the tool Debezium) simply tails that same log, turns each committed change into an event, and replays it into the target. It's efficient — it adds almost no load because it runs no extra queries — and it captures everything, including deletes, in commit order. But it's fragile: the log format isn't standardized and shifts across engine upgrades, and a schema change on the source can break the pipeline.

CDC is one of the most boring but most fundamental operations powering modern society — but it's so brittle we joke it should be called "continuous data corruption."

A schema change on the OLTP side breaks the pipeline; data engineers get woken at 3 a.m. ("everyone became a $5B company just doing CDC.")

HTAP → L-TAP: unify the storage, not the query engine

Concept — HTAP, the "holy grail"

HTAP (Hybrid Transactional/Analytical Processing — a term Gartner coined in 2014) is the long-sought dream of one database that serves both live transactions and analytics on the same fresh data. The appeal is obvious: no second system, no CDC pipeline, no lag — you can analyze data the instant it's written. The catch is that "great at fast small writes" and "great at big scans" pull the storage engine in opposite directions (recall row vs column above), so a single-engine HTAP system tends to compromise on both — and, as the Databricks folks add, you also give up the mature Postgres and Spark ecosystems for a smaller proprietary API you have to build from scratch.

Databricks's alternative rests on two background ideas worth knowing first.

Concept — separation of storage & compute, and open formats

Separation of storage and compute is the cloud-data pattern (pioneered by Snowflake) of keeping the data in cheap, shared object storage — Amazon S3 and its kind: virtually unlimited, highly durable, dirt-cheap file storage accessed over a simple HTTP API — while spinning processing power up and down separately. The point is independent scaling: store a petabyte cheaply and pay for compute only when a query actually runs, instead of the old model where storage and compute were bolted together and had to grow in lockstep. Lakebase applies this to Postgres (it came largely from Databricks' acquisition of Neon, a "serverless Postgres" company).

Open table formats are the other piece. Analytics data sits in the object store as Parquet files — an open, compressed, columnar file format. On top of those files, Delta Lake and Apache Iceberg are open table formats that add the guarantees a raw pile of files lacks: ACID transactions (no half-written/corrupt data), schema evolution, and "time travel" to past versions — via a metadata/transaction-log layer over the files. This is the "lakehouse" idea: cheap data-lake storage plus warehouse-grade guarantees on top. "Open" is the load-bearing word — no vendor owns the format, so many engines (Spark, Trino, DuckDB, even Snowflake) can read and write the same files, and your data isn't trapped in one product.

Dream Engine: an ML "factory" for building a database

Concept — what a database engine has to decide

For any given query there are usually many ways to execute it — different algorithms and data structures (which join method, whether to build a hash table or sort, which index or scan). A database's query optimizer normally picks among them using hand-written rules and cost models. The best choice depends on the data and the query, and the choices genuinely trade off against each other — an algorithm that's fastest for a tiny low-latency lookup can be terrible for scanning petabytes. Dream Engine's bet is to replace that hand-tuned judgment with a learned one.

Every analytics engine is ~a decade old — started narrow, then accreted hacks into "a gigantic pile." Rewriting from scratch risks second-system syndrome (the classic trap where an engineer's over-ambitious second design collapses under its own scope). So instead of hand-picking algorithms from academic papers (which "look great on 70% of workloads, backfire on 30%"), they built a factory.

This is the clearest instance of the episode's macro point: because Databricks owns a decade of query traces and table layouts, it can build a new engine fast and be confident it's good. Category footnote — vector DBs (Turbopuffer, Chroma) are collapsing into general blob storage: "a vector database should never have been a separate category."

Strategy & the macro bet

Open vs proprietary. Open-source a layer when it benefits from network effects (people write connectors/libraries you'd never have time to build); keep proprietary what must be a service (an ops team so your streaming jobs don't lose data at 3 a.m.). Databricks "never had a proprietary format" (Parquet → Delta → Iceberg) — cited as the core reason it outpaced Snowflake (its main cloud-data-warehouse rival; open + starting upstream at large-scale ingest, vs proprietary fast-serving of small data).
Design protocols even in the vibe-coding era. When multiple parties moving at different speeds must coordinate (agents↔agents↔tools, or cross-company data sharing), you still design real protocols/APIs — analogous to internet/network protocols. See Open/Delta Sharing (a live table view shared across companies) and clean rooms (share results without exposing rows).
Ship incrementally; don't boil the ocean. Products built "in weeks." First question to any team: "Who's the target customer — are you on a first-name basis, texting them?" Overfitting to 1–2 customers is fine (downside small, upside big); the real risk is being so ambitious you have no customer.
Customizing models keeps getting easier. Smarter base models → better traces → RL (learn from your own past traces) + synthetic data. They report a pipeline where an open model generates its own training environments, trains itself, and beats Opus / GPT-5.5 on a task. Example win: a specialized document-parsing vision model ~100× cheaper than frontier models and more accurate. Plus advisor models — small, cheap models that coach a bigger agent on one narrow part of a task — and specialized sub-agents.

Concept — RL fine-tuning & synthetic data

A model is first pre-trained to predict text; you then fine-tune it for a specific job. RL (reinforcement learning) fine-tuning does this with a reward signal instead of fixed answers — the model tries something, gets scored good/bad, and is nudged toward higher-scoring behavior (the classic version, RLHF, learns the reward from human preference rankings; newer "verifiable-reward" variants score against an objective check, like whether generated code passes its tests). Synthetic data is training data generated by a model rather than collected from the real world — cheaper and faster, and useful for targeting a narrow skill. Put together, that's how a smaller, cheaper, specialized model can beat a giant general one on one task: better base models now produce good enough traces and synthetic examples to train the specialist. (The main risk to know: training round after round on purely model-made data can cause "model collapse," where quality and diversity quietly degrade — so it's mixed with real data.)

Data is the new oil — context is the new oil — tokens as IP.

They explicitly connect this to Satya Nadella's "frontier ecosystem" essay ("he sounded like a Databricks CEO"): any technology that makes your proprietary data more actionable gives an advantage, and AI is the latest such technology — which is exactly why owning your data/traces (and the learning loop over them) is the durable moat.
