Swarm Management of Agent Harnesses

By Aparna Dhinakaran — Co-founder & CPO, Arize AI • Published May 4, 2026 • Saved May 31, 2026
AI Agents Swarm / Orchestration Long-Running Agents Control Plane

Most frameworks have crossed the first line: they can spawn subagents. That is not swarm management — it's the beginning of the problem. The interesting question is what happens after the child exists.

The core distinction

Delegation asks: how does one agent split work? Swarm management asks: how does a runtime own many agents over time? A harness is a loop over tools. A swarm manager is a loop over running harnesses, making sure they progress. This is the next layer above the agent harness.

Picks up exactly where Hermes Harness Architecture left off (delegation → first-class orchestration) and extends component #4 of What is an Agent Harness. Reads OpenClaw and Hermes as the worked examples.

Hermes — great delegation

delegate_task creates child AIAgent instances, runs them in parallel, streams progress, applies timeouts, interrupts, returns structured summaries. Clean and useful — but the child lives inside the parent tool call.

OpenClaw — real swarm management

Subagents become gateway sessions: durable session keys, run IDs, lifecycle records, parent-child lineage, cleanup policy, and a push-based completion path back to the requester.

The seven requirements of swarm management

1 · The agent needs durable identity

If you cannot address a child, you cannot manage it — you can only wait for it or cancel a local future. In OpenClaw a spawned child gets a session key plus a run ID:

agent:<targetAgentId>:subagent:<uuid> ← session key (WHERE the child lives) + run ID ← names the CURRENT execution You need both.

That turns the child into something the gateway can list, patch, delete, and link back to a parent — sitting next to normal chat, cron, and ACP sessions. The runtime tracks: session key, run ID, requester, controller, depth, role, workspace, task label, cleanup policy, timestamps, outcome.

If those answers only live inside a model's context window, the runtime cannot manage the swarm.

2 · Completion is routed, not returned

The simple delegation contract (parent calls → child runs → parent blocks → child returns → parent continues) breaks down once the parent isn't just waiting on the same call stack. In a real swarm the parent may be active, idle, another subagent, restarted, or gone. OpenClaw uses a push-based model: sessions_spawn returns acceptance + bookkeeping; the result arrives later through the registry and announce flow.

3 · Queues matter more than prompts

Once agents can spawn agents, the runtime must handle concurrency practically: strict ordering within a session (two messages shouldn't collide in one active loop) and parallelism across sessions (or the fleet becomes single-file). This is why swarm management looks less like a framework and more like runtime infrastructure.

5 · Roles are a runtime safety mechanism

Flat swarms don't scale — if every child can spawn every other child indefinitely, the system becomes useless or dangerous. The simple split: orchestrators coordinate (spawn workers, inspect status, synthesize); leaves do work and cannot coordinate further. This should be enforced in tools, not suggested in prompts.

6 · Recovery keeps the runtime in control

A swarm manager cannot fire-and-forget. If it owns children, it must know when they are stuck, missing, orphaned, completed, or completed-but-undelivered. OpenClaw's subagent registry listens to lifecycle events, uses agent.wait, persists to disk, restores runs on startup, and runs retry timers + grace windows. And it runs a sweeper:

The sweeper — "not glamorous, but it makes the system real"

  • Checks active runs without live run context; reconciles against session state
  • Expires old session-mode records after cleanup
  • Removes or retries stuck pending lifecycle states
  • Marks delivery failed instead of leaving cleanup half-done forever

This is operating-system work. A swarm manager needs something close to a process table — what's running, who owns it, what happens when it exits, which children die with the parent, which sessions survive restart, which outputs haven't been delivered.

7 · Agent state outlives the run

Every swarm manager eventually becomes a cleanup system. Subagents create transcripts, run records, browser sessions, MCP runtimes, attachment dirs, workspace files, delivery status, lifecycle hooks, cost metadata. Someone must own that after the model stops thinking about it.

Bounded delegation

Cleanup is local: the child object closes, the thread exits, the parent gets JSON.

Swarm management

Cleanup is distributed: child may be a session; run done but delivery incomplete; session kept but attachments removed; parent may need one more wake-up after descendants settle. Cleanup is a state machine.

Swarm management is the layer above the agent

OpenClaw is interesting because there's no magical SwarmManager object. The swarm emerges from ordinary runtime machinery:

session keys · lanes · run IDs · registry records · lifecycle events internal completion events · queue policy · delivery routing cleanup decisions · recovery sweeps

The harness question was "can an agent call tools?" The next question is where agents live, who owns them, how they report back, how they're stopped, and what survives after restart. That is the layer that turns "a single agent harness" into a fleet of well-coordinated agents.
