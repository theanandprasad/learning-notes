How to Build Claude Subagents Better Than 99% of People

By Jey (@Jeyxbt) • Published Jun 18, 2026 • Saved Jun 18, 2026
Claude Code Sub-agents Context Engineering Orchestration

One Claude talks to you. Five more can be working behind it on five different jobs, and you never see a single one. The whole game is that the heavy work stops happening in the window you're sitting in.

What a sub-agent is

A sub-agent is a second Claude the main chat hires for one job — its own context window, its own model, its own permissions, running in parallel. It does the messy work somewhere else and hands back one clean answer. The orchestrator is the only thing that talks to you; sub-agents answer to it, not to you and not to each other. "The win was never that Claude does more work. It is that the heavy work stops happening in the window you are sitting in."

The receipt that makes the rest matter

In one real run, a reviewer sub-agent burned 22.8K tokens reading a plan and tearing it apart. What came back to the main session was a few lines. "The main context never saw the 22.8K. It saw the verdict." Your main window fills as you work; the more it fills, the more polluted it gets, and the dumber the model runs for the rest of the chat. A sub-agent sidesteps that.

The fleet economics

The anatomy: a custom sub-agent is one Markdown file

Those five-persona demos (beginner / engineer / business owner / publisher reading the same draft) are just the built-in general-purpose agent prompted five ways. A custom sub-agent is something you write — and it's smaller than people expect: a single Markdown file with YAML front matter in .claude/agents/. Structurally it's almost the same object as a skill file; the only real differences are the clean context window, the parallel run, and the model it sits on.

The 12 levers nobody tells you

1 · The description is the trigger. When Claude decides whether to fire a sub-agent it reads the front matter and nothing else — the name and description, not the body (progressive disclosure). So the description is the agent. Tighten it until the trigger is unambiguous, then test in the wild; when it misbehaves, ask "Why didn't it fire?" and sharpen.
2 · Keep the description short. The front matter is read on every routing decision, so a fat description is a tax Claude pays again and again. (The "generate with Claude" option hands you one three sizes too big — trim it hard.)
3 · Close your quotes. A single unclosed quote in the YAML kills the trigger silently — no error, no warning, it just never fires. Before debugging logic, check the front matter is even valid.
4 · Enforce read-only at the tool layer. "Asking nicely is not a permission system." Writing "please don't modify files" in the prompt builds a suggestion. Real safety lives in the tools field — list only Read, Grep, Glob and the agent physically cannot touch your data. "If your AI can touch data, assume it will. Build the wall, don't ask politely."
5 · Verify any agent file you didn't write. A sub-agent is just Markdown — so you can download one, and you can download a prompt injection. Point a read-only verifier at any file you didn't write and scan for injection or privilege escalation before it joins the fleet.
6 · Cap the loop with max_turns. A sub-agent can get stuck circling — re-reading, re-trying, never landing. max_turns stops it after N exchanges and reports what it has. Cheap insurance against a runaway.
7 · Match the model to the task. Running the best model on everything is how you set money on fire. Model is a per-agent field — orchestrator on Opus (it plans/decides), research worker on Haiku (it reads/summarizes). Only let a worker inherit the parent model when the job genuinely needs it.
8 · Pick your invocation path on purpose. Four ways a sub-agent fires: Automatic (Claude reads the description and decides), Proactive ("use proactively" in the description → jumps in unasked), Explicit (name it: "use the plan-roaster sub-agent"), Direct launch (start a session as it from the CLI). Proactive is the one people miss.
9 · Project-level for the repo, global for you. Project agents live in the repo's .claude/agents/ and ship with the code; global agents live at user level, follow you everywhere, stay private. Promoting one is just moving a file — build it global while you tune, drop it in the repo once it earns a spot.
10 · Skills and sub-agents compose, they don't compete. A skill runs inside the main session; a sub-agent runs in a clean, isolated, parallel session on its own model. That's the only real difference — so they stack (skills can call sub-agents, sub-agents can call skills). Skill = work belongs in the main thread; sub-agent = you want it out of it.
11 · Dynamic workflows can spin up a swarm. On the newest Claude, one orchestration can run sub-agents in parallel — 3 in one test, 40 in another, 210 in a single run — for order-independent work. It chews through limits fast; the trigger word was deliberately renamed so it wouldn't kick off by accident. Reach for the swarm when the parallelism is real, not as a reflex.
12 · The default general-purpose agent is fine — until you've done the job twice. Repetition is the signal. "A narrow specialist you reuse beats a generic agent you re-brief every single time."

When to delegate — and when it quietly hurts you

This is where fleets fall apart: people force sub-agents onto work that's wrong for them, and the agents make everything slower and pricier. The one question that covers most of it: "Is this about to dump a pile of output into my chat that I'll never read again? If yes, send it out."

The reviewer trick that pays for itself

By default, an AI is a sycophant. Ask the same session that helped you build a plan whether the plan is good, and it leans toward yes — it's already invested. A fresh-context reviewer carries none of that: it never watched you build the plan, so it has nothing to defend.

The clean context is the objectivity. You can't get an honest second opinion from the session that wrote the first one.

That's why the plan-roaster runs read-only with one job: find the weakest assumption and say it cold. The same logic scales to a security auditor, a test writer, a docs writer, a database specialist — each a narrow silo, great at its single thing, blind to your ego about the rest. That's the assembly line.

The shape of it

Stop picturing a sub-agent as a smarter Claude. Picture a clean room with one door, one worker, one job.
The bottleneck was never how smart the model is. It was how clearly you can divide the work.
