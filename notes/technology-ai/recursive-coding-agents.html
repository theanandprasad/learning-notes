Recursive Coding Agents

By Raymond Whitcomb (OpenProse / independent research) · conference talk • Published 2025–26 • Saved Jul 7, 2026
Recursive Language Models Coding Agents Orchestration OpenProse

The bottleneck to trustworthy AI agents isn't intelligence — the models already know the whole internet. It's that they can't reliably deliver outcomes. The fix isn't a smarter model; it's better orchestration — and "recursive language models" are Whitcomb's bet on how.

The thesis — "mismanaged geniuses"

Today's agents are mismanaged geniuses (a phrase from the RLM authors): the raw intelligence is there, and the missing layer is how we specify, manage, reuse, and verify the work. Trust is reliability — you can't trust something that isn't reliable — and the next step forward is behavioral/orchestration, not more raw IQ. The tool this talk champions: Recursive Language Models (RLMs), and applying them to coding agents.

The problem: agents you can't trust

We all want reliable co-workers getting things done while we're out on a hike. But the models "can't reliably deliver outcomes, so I can't trust them." Whitcomb's illustrative war story: one day a near-working SaaS app from a single (long) prompt; the next, "Claude Code emptied the entire contents of my Solana wallet." That whiplash is the whole problem — the intelligence is there, the management isn't.

Concept — Recursive Language Models (RLMs)

What. RLMs (introduced Oct 2025 by Alex Zhang, Tim Kraska & Omar Khattab at MIT) are an inference strategy, not a new model architecture. The slogan: "the context itself is the object of computation." Instead of the whole prompt being read into the model's context window, the full prompt is treated as a variable — it might be a file, or many files — sitting in an executable environment.

Why. A model's context window is a hard ceiling, and stuffing a huge prompt into it also degrades quality. RLMs sidestep both: because the prompt lives as data in a REPL, the model can handle inputs 1–2 orders of magnitude larger than its context window (tens of millions of tokens) and often answer better than a plain long-context model — at comparable or cheaper cost.

How. The model is told to explore the prompt symbolically inside a REPL (Python in the original paper) — grep it, slice it, sample it — rather than read it all. And it can recursively call other LLMs / sub-RLMs on the pieces, decompose the problem, and work its way back up to a final answer. In the reference implementation, GPT-5 / GPT-5-mini is queried inside a Python REPL that holds the user's prompt in a variable.

Why they're so powerful (the results)

Concept — DSPy & ARC-AGI-3 (the tools/benchmarks below)

DSPy ("Declarative Self-improving Python", from Omar Khattab, NeurIPS 2023) is a framework for programming, not prompting, language models: you write modular Python and DSPy compiles/optimizes the prompts under the hood, so pipelines are robust instead of brittle prompt strings. It's the ecosystem the RLM benchmark work is built in.

ARC-AGI-3 (ARC Prize Foundation, launched March 2026) is the first interactive reasoning benchmark: agents must explore unfamiliar game-like environments, form a model, and plan — learning on the fly with no instructions. At launch every frontier model scored well under 1% while humans solved 100% — which is what makes the RLM result below notable.

What counts as an RLM — the rubric

To cut through "isn't this just X?", Whitcomb offers a rubric (companion GitHub repo). An RLM needs all of:

1 · An executable environment — a REPL the model can run code in.
2 · The prompt is externalized — it lives as a variable/file, not jammed into the context window.
3 · Code is what calls the model — the program drives the LLM, not the other way round.
4 · The model picks the decomposition — it decides how to break the problem into sub-calls/sub-agents.
5 · State stays symbolic — work over references to the data, not the raw data pulled into context.

From RLMs to recursive coding agents

Swap "RLM/LLM" for "agent/sub-agent" and you get the same picture — a recursive coding agent. (Trick question: an RLM already is one.) The interesting work is applying RLM principles to real coding agents. Whitcomb's experiments:

RLM-as-a-tool. Wrap the RLM package as a CLI so a coding agent can call an RLM as a single tool — e.g. "go sift this 100M-token corpus," and the RLM does the RLM thing.
Y-Pi — a fully recursive harness. Make the coding agent call itself. Built on Pi (a minimal, extension-first coding agent by "Mario"); "Y" is the λ-calculus Y combinator (recursion). Now shipped as a pure pi-recursive extension + the y-pi convenience wrapper — "Pi calls Pi calls Pi," depth of your choosing.

Notable projects in the space: the original RLM implementation (Alex Zhang) and its DSPy version (his benchmarking go-to); AXE (agent-native — an AXE agent can write a whole TypeScript interface to another AXE agent, recursion all the way down); the Unix RLM (Dan @ OpenProse — pure bash, the environment is just the Linux file system, showing "the REPL can be anything"); and OpenProse itself.

Is Claude Code an RLM?

The most-asked question. Release-day answer: no. Now, arguably yes — because of Dynamic Workflows. As Omar Khattab put it: "Claude Code is finally an RLM now that you have dynamic workflows."

Concept — Claude Code Dynamic Workflows

Dynamic Workflows let Claude Code turn "the plan" into actual orchestration code that spawns and coordinates agents (spawn one, run several in parallel, pipeline them), rather than doing everything in one chat thread. That's exactly what makes it recursive/RLM-shaped: code decides the decomposition and calls the model over pieces. Whitcomb points to Anthropic's write-up "A Harness for Every Task" (six workflow patterns). For the deeper mechanics, see the sibling note The Plan Is Code.

To show the rubric in action he wrote two Claude Code workflows: one deliberately not an RLM (a hard-coded map-reduce), and one that is — "deep research over a file system" (pick a handle, hand it to an agent, analyze, bring findings back).

OpenProse: turn any coding agent into an RLM

Dynamic Workflows are great but Claude-Code-only. OpenProse is the portable alternative. It's "technically a programming language, but not compiled by your computer — compiled by your coding agent": a Markdown spec in logical English, no exotic syntax. A prose write command gets Claude Code / Codex / Pi to author a .prose.md file for you (much like Claude Code's ultra-code command writing workflows).

Golden sessions

Golden sessions close the loop back to the trust problem. You have one great day with an agent — then want that quality on demand. OpenProse can take a "golden session" (Claude Code, Codex, Pi…), have an agent deconstruct it, and turn it into a reusable Prose workflow — so you reach that reliable "golden state" over and over.

Three takeaways

1 · Trust is reliability

The next step isn't more raw intelligence — it's behavioral: orchestration, specification, verification.

2 · RLMs = a new test-time paradigm

Tool calling and reasoning unified — you reason through tool calls, and one tool is "call another agent to do a subtask," recursively.

3 · Coding agents can be RLMs — but aren't automatically

Claude Code Dynamic Workflows and OpenProse are two ways to turn a coding agent into a recursive one.

With great power comes great responsibility. Until next time — recurse responsibly.
