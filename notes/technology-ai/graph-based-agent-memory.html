Graph-Based Agent Memory

By Neo Kim · The System Design Newsletter • Published Jul 7, 2026 • Saved Jul 7, 2026
Agent Memory Knowledge Graph Object Storage Git for Data

You have several agents on one project — coding agents in your terminal, a chatbot in the browser, a background agent on AWS — and each keeps its own memory. So you copy decisions between chats by hand. What's missing is one shared memory every agent can read from and write to — and building that turns out to be a hard systems problem.

ℹ️ Sponsored/partnered piece. The author partnered with Omnigraph (an MIT-licensed open-source graph engine) and uses it as the case study. The problem framing and design patterns are broadly educational and vendor-neutral; the specific product is the sponsor's. Ends with paid-masterclass and referral CTAs.

The core idea

A shared agent memory must do three jobs: read (assemble context by following connected facts), write (persist knowledge that outlives the task), and — the genuinely new one — coordinate (let many agents read/write at once without overwriting each other or exposing half-finished, possibly-wrong changes). A pile of documents or a vector database handles read and write, but not coordination. The answer explored here: a typed graph on shared object storage, where every change is proposed on a branch, reviewed, and merged — "Git for AI memory."

Why the obvious approaches break

Reads and writes aren't new… but coordination is. Traditional memory assumes a few deterministic writers making short, careful updates. AI agents reason for minutes, revise conclusions, make mistakes, and run alongside many others touching the same knowledge.

A graph alone isn't enough — you need a schema

Let many agents write freely and a subtle failure appears. Three agents record the same event as "deploy," "deployment," and "release" → three different nodes, because nothing tells the graph they mean the same thing. Ask "show every deploy this week," and the query follows only the deploy nodes — silently missing the rest.

The result looks complete… but it's WRONG. The graph didn't fail — the schema did.

So define the schema before anyone writes: which node types may exist, which relationships connect them, and how each node is identified (a @key). Every agent writes against the same schema, so hundreds of agents describe the world the same way. In Omnigraph the schema is a plain-text file the agent generates and updates from your graph — e.g. nodes Person/Service/Deploy, and an edge Deploy -> Service meaning "from the deploy to the service it touched." The memory isn't just a graph — it's a typed graph with a shared schema.

Where the graph lives: object storage

The graph lives in object storage (a cloud data bucket — S3, GCS, R2). Why S3:

Low cost
a few cents/GB/month — memory grows without a big DB cluster
High durability
replicated across many availability zones; it's the system of record
Shared access
not tied to one machine — every agent with credentials reads/writes one bucket

The catch — no atomic writes. Object storage safely stores one object at a time, but can't commit several as a single all-or-nothing operation (what a database calls an atomic write). Write A, then B, then C, and the bucket treats them as three independent writes — there's no "commit all three or none." And a graph update almost always touches many objects. So how do you get atomicity from storage that doesn't offer it?

Atomic writes via a manifest

The graph is not one file. Omnigraph splits it into many stores — one per node type and per edge type — each keeping every record of that type in versioned files (columnar format Lance under the hood), plus a tiny manifest store. Recording "Alice pushed deploy d-913 to checkout" writes three stores: nodes/Deploy/, nodes/Person/, and edges/TriggeredBy/. Crash mid-way and the graph is inconsistent — and the bucket can't roll back, because it never knew those writes belonged together.

The manifest fixes this. It's a small table recording, for each store, which version is live right now, under two rules: (1) stores are immutable — a write creates a new version beside the old, which stays live until the manifest says otherwise; (2) each store has exactly one live version — the manifest is the pointer.

Compare-and-Swap (CAS) prevents lost updates

Before updating the manifest, Omnigraph checks whether every store is still on the version the writer originally read. Writer started from Deploy v7 and made v8 → "Is Deploy still v7?" Yes → the commit succeeds. No → someone already published a newer version, so the commit is rejected and the writer retries from the latest state. The manifest is append-only. CAS stops one writer from silently clobbering changes it never saw.

Git for AI memory: branch, review, merge

CAS stops conflicting writes — but it doesn't judge whether a write is correct. Agents hallucinate; a wrong write that wins CAS instantly becomes shared fact. So Omnigraph borrows the Git workflow: don't edit shared truth directly — branch, change, review, merge.

Merge conflicts. Changes to different parts of the graph merge automatically (both kept). A real conflict is only when two branches set the same field differently (A: d-913 → rolled-back; B: same field → succeeded) — Omnigraph won't guess; it reports the exact node, field, and both values for a human/policy to resolve. Graphs add a second kind: one branch deletes a node while another adds an edge to it (a dangling edge) — detected before the merge. "Every disagreement becomes an explicit conflict, not a silent overwrite." (Prior art: Dolt = Git for SQL, lakeFS = for data lakes; Omnigraph extends it to typed graphs.)

Who did what: commits, authorization, audit

A safe write leaves three questions — who's allowed to change this, who actually did, and how to undo it — all answered by one object: the commit.

Authorization lives in the engine, not the API. You can reach Omnigraph via HTTP, a CLI, or an embedded SDK (agents usually connect through an MCP server). Enforcing permissions at the HTTP layer would leave the CLI and SDK as back doors — so every write passes the same authorization check inside the engine. No path bypasses it.
Policies in Cedar. Rules (written in Cedar, AWS's open policy language) govern the actions: read, make a change, create/modify a branch, merge into main, change the schema. Permissions are branch-level today; field-level ("edit a deploy's status without touching its owner") is on the roadmap.
Identity comes from the token, not the request. Each actor (agent, human, service) gets a token; the engine identifies it from the token. A request can claim "I'm an admin" — the token decides.
Commit history = the audit trail. Each accepted change is a commit recording who, when, and its parent — chained into the full history. No separate audit log needed. Crash-recovery commits use a reserved system identity, so you can tell human vs agent vs system apart.

Everything is reversible. Because nothing is ever updated in place — every change is a new version beside the old — you can read the graph exactly as it looked before any merge, compare, or roll back (discard a branch, or revert a merge with a new commit restoring an earlier state). Nothing is reconstructed because nothing was overwritten.
