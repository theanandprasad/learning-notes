Agentic Memory (for TanStack AI)

Blue Collar Coder (YouTube) · demo & explainer • Published 2026 • Saved Jul 7, 2026
Agentic Memory Vector Search TanStack AI LLM Infrastructure

Tell a chatbot "I like sushi," and later it suggests sushi for dinner — great. Start a new conversation, and it's forgotten you entirely. This is a walkthrough of a prototype that gives an agent durable memory across chats, and the surprisingly small set of moving parts behind it.

The core idea

By default a chat only "remembers" what's in the current conversation — so it forgets everything the moment you start a new one. Agentic memory fixes this by extracting durable facts from your chats and reusing them: before each turn it recalls facts relevant to what you just said and injects them into context; after each turn it retains new facts from the transcript. This POC (built for TanStack AI) wraps that behind one generic MemoryDriver interface, over three vendors (Hindsight, Mem0, Honcho) or a DIY local store.

Concept — TanStack AI

TanStack AI is a type-safe, provider-agnostic TypeScript AI SDK — a unified way to build streaming chat, tool-calling, and agents across OpenAI, Anthropic, Gemini, etc., and across React/Vue/Svelte/Solid. It's pitched as "the Switzerland of AI tooling" (vendor-neutral). This note's project is a proof-of-concept exploring how to add memory to it — so "for TanStack AI" just means it plugs in as middleware there; the ideas apply to any agent.

The demo: why agents forget

Concept — the types of LLM memory

The project borrows a tidy taxonomy (from cognitive science) for what "memory" can even mean for an LLM: model memory = what it learned in training; working memory = what it holds within a single response; episodic / context-window memory = the current chat; and long-term user/application memory = durable facts, preferences, observations, and summaries gleaned across conversations. That last one — persistent and personal — is what "agentic memory" means here, and the only one you have to build yourself.

What a memory system actually does

Behind the generic MemoryDriver interface, a memory system is just three operations:

Concept — embeddings & vector search (how "recall" finds the right fact)

The problem: when the user says "what's for dinner?", how does the system know the fact "user likes sushi" is relevant? Plain keyword matching fails — the words don't overlap.

The mechanism: an embedding model turns each fact into a vector (a list of numbers capturing its meaning), stored in a database. At recall time the query is embedded too, and the system does a vector search — returning the stored facts whose vectors are closest in meaning. This is "concept search, not keyword search," which is what an LLM wants: "dinner" can surface "likes sushi" because they're semantically near, even with no shared words.

The DIY pipeline (how the local version works)

The four memory engines

Concept — the memory vendors

Mem0 — a scalable long-term-memory layer for agents; incrementally extracts salient facts and manages them (add / update / delete), as a managed API or self-hosted. Honcho — a managed memory service that abstracts away the embedding store, retrieval, and persistence so you add "remembering" with minimal code. Hindsight — the most distinct: it maintains graphs of facts, experiences, entities, and beliefs with retain / recall / reflect operations, aiming at reflection and self-improvement over time (and, per the talk, has tool support). All three convert conversations into reusable structured memory (facts, preferences, entities).

Setup is light: you need an Anthropic key + an OpenAI key, but not paid subscriptions to the three vendors — they run locally via Docker (though some still call out to LLMs, so the keys are needed regardless).

Adding it to a chat (the code)

Two imports (from the unpublished ai-memory package) do the work: a createMemoryMiddleware factory and an engine (e.g. hindsightEngine). You give it the engine and a scope (who you are + which session), then drop the returned middleware into your pipeline — it puts itself in the right place to recall before the turn and retain after.

Two memory "roles" — full vs incognito

You choose the role memory plays in a conversation. Full = recall + retain (uses what it knows about you and stores new facts from this chat). Incognito = it can use what it already knows about you, but won't store anything from this conversation — "know about me, but don't remember this chat." (The talk labels this "retain-only"; the described behavior is recall-without-storing — useful for sensitive sessions.)

Beyond TanStack AI: memory for any coding agent

The same pattern works if you run coding agents agnostically (Claude Code, Codex): there's no middleware hook, so you drive the two phases yourself.

Episodic memory is the chat you're in. Long-term memory is what makes an agent feel like it actually knows you.
