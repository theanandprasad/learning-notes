Prompt Caching Is Everything (Lessons from Building Claude Code)

By Thariq (@trq212, Anthropic) • Published Feb 20, 2026 • Saved Jun 27, 2026
Claude Code Prompt Caching Latency & Cost Agent Harness

There's an old saying in engineering: cache is everything. For long-running agents, it's not a metaphor — it's the load-bearing wall. Claude Code is built around prompt caching, top to bottom.

The thesis

"Cache is everything" in engineering — and the same holds for agents. Long-running agentic products like Claude Code are made feasible by prompt caching: reusing computation from previous roundtrips to cut latency and cost. Claude Code builds its entire harness around prompt caching — a high cache hit rate lowers cost and funds more generous rate limits, so the team runs alerts on the cache hit rate and declares SEVs (incidents) when it drops too low.

An agent makes many roundtrips to the model over a single task. Each roundtrip resends the entire conversation so far — system prompt, tool definitions, every prior message and tool result. Without caching, the model re-processes all of it from scratch every single turn. With caching, the model reuses the work it already did on the unchanged part of the request. For a 50-turn coding session, that's the difference between viable and unaffordable.

The core mechanic: prefix matching

Prompt caching works by prefix matching. The API caches everything from the start of the request up to each cache_control breakpoint. On the next request, if the prefix is byte-for-byte identical, the cached computation is reused — and you only pay full price for what comes after the last match.

This has one overwhelming consequence: order matters enormously. Put the most stable content first and the most dynamic content last. Anything that changes invalidates the cache for everything after it.

The non-obvious part: tools are part of the cached prefix. They sit at the very front of the request, ahead of the system prompt and the conversation. So adding or removing a single tool invalidates the cache for the entire conversation — even a 40-turn one. This is the trap most agent builders walk into, and it reframes how you should design state changes.

Design example 1: Plan Mode — design around the cache

Plan Mode is a feature where Claude Code explores the codebase and proposes a plan without editing files. The intuitive implementation: when the user enters plan mode, swap the tool set to only read-only tools. That feels clean — and it breaks the cache. Changing the tool list invalidates the prefix, so every plan-mode toggle reprocesses the whole conversation.

The intuitive (wrong) way

On entering plan mode, replace the tool set with read-only tools. Clean mental model — but it mutates the prefix and breaks the cache on every toggle.

The cache-safe way

Keep all tools in the request at all times. Model plan mode as tools: EnterPlanMode / ExitPlanMode. Toggling injects a system message, not a tool change.

The cache-safe design keeps every tool in the request at all times and models plan mode as tools: EnterPlanMode and ExitPlanMode. Toggling plan mode just injects a system message into the conversation — "you're in plan mode: explore the codebase, don't edit files, call ExitPlanMode when the plan is complete." The tool definitions never change, so the prefix stays intact and the cache holds.

The bonus. Because EnterPlanMode is a tool the model can call itself, Claude can autonomously enter plan mode when it detects a hard problem — pausing to think and plan before acting — with no cache break. Modeling state as a tool didn't just protect the cache; it handed the capability to the model.

Design example 2: Tool Search — defer instead of remove

The same principle generalizes. As an agent gains tools, you might be tempted to trim the tool list down to only what's relevant for the current task — removing the rest to keep the model focused. But removing tools breaks the prefix, exactly like the naive plan-mode design.

The fix is Tool Search: rather than removing tools, defer tool loading. The tools that aren't needed yet are loaded lazily on demand instead of being stripped out, so the cached prefix stays intact. You get the focus benefit without paying the cache-invalidation cost.

Remove unused tools

Trims the prompt — but every removal rewrites the prefix and invalidates the cache.

Defer tool loading

Load tools lazily when needed. The prefix is never mutated, so the cache survives.

Lessons learned

The whole approach distills into five rules. Each is a direct consequence of the prefix-matching mechanic.

Prompt caching is a prefix match. Any change anywhere in the prefix invalidates everything after it. Design your whole system around this constraint — get the ordering right (stable first, dynamic last) and most caching works for free.
Use messages, not system-prompt changes. Tempted to edit the system prompt — to enter plan mode, change the date, and so on? Insert these into messages during the conversation instead. The system prompt is part of the prefix; the tail of the message stream is not.
Don't change tools or models mid-conversation. Use tools to model state transitions (like plan mode) rather than changing the tool set; defer tool loading instead of removing tools. Switching models mid-conversation invalidates the cache for the same reason.
Monitor your cache hit rate like uptime. Alert on cache breaks and treat them as incidents — a few percentage points of cache-miss rate can dramatically affect cost and latency. Claude Code declares SEVs when the hit rate drops.
Fork operations must share the parent's prefix. For a side computation — compaction, summarization, skill execution — use identical cache-safe parameters so you still get cache hits on the parent's prefix, rather than starting a cold, uncached request.
Claude Code is built around prompt caching from day one — you should do the same if you're building an agent.
