What is an Agent Harness?

By Aparna Dhinakaran — Co-founder & CPO, Arize AI • Published Apr 24, 2026 • Saved May 31, 2026
AI Agents LLM Infrastructure Architecture Agent Engineering

"Can anyone actually tell me what a harness really is?" — asked, with real skepticism, at a hacker event. We all use the word; few can define it. Here is the answer.

The one-line definition

A harness is the fixed, out-of-the-box architecture — born from real coding agents — that wraps a language model in a feedback loop so it can act, observe the consequences, and adjust until a problem is actually solved. A model alone is a one-shot text generator; a model in a harness is an agent.

What a harness is not

Frameworks like LangChain and LangGraph are not harnesses. They are tools designed for humans to build agents: abstractions, configuration options, dozens of knobs, and a lot of rope. You wire together chains, define state graphs, pick a retriever, configure memory. The fundamental assumption is that a human architect will configure these pieces correctly.

The article pushes back on the framing "if you're not the model, you're the harness" — calling it a weak attempt to re-label earlier 1.0 agent frameworks with the new "harness" terminology.

Framework (e.g. LangGraph)

  • Built top-down from abstractions
  • A human assembles the agent
  • Needs configuration to exist
  • A human-configured state graph

Harness (e.g. Claude Code, Cursor)

  • Born bottom-up from real coding agents
  • The agent accomplishes the task
  • Works out of the box, no assembly
  • A while-loop over tools

What an agent harness is

The modern harness was born bottom-up, out of coding agents solving real problems. Cursor, Claude Code, Windsurf, and Codex are all harnesses. Each started from one concrete problem — make an LLM write and edit real code across real repos — and independently converged on remarkably similar architectures: a while loop that calls tools, a context manager that compresses history, a permission layer that keeps things safe. Arize hit the same patterns building Alyx, their in-product agent. Different domains, same bones. That convergence is the signal.

Two things make a harness fundamentally different

1. It works out of the box. You don't configure a harness into existence — it ships as a working agent with a fixed architecture (loop + context management + tool registry + permission layer), all wired together and already running. No assembly step.

2. It is not for humans to build agents. It is designed for the agent to accomplish almost any task. The model reads instruction files, discovers and composes tools, writes its own skills, and spawns subagents. The human provides the goal; the harness figures out the rest.

The nine components of a modern harness

Just as computer architectures matured in the 1980s, harness architectures are expected to mature over the next couple of years. The "Harness 1.0" architecture has nine component areas:

  1. Outer iteration loop
  2. Context management & compression
  3. Managing Agent Skills and tools
  4. Managing subagents
  5. Built-in / prepackaged skills
  6. Session persistence and recovery
  7. System prompt assembly & project context injection
  8. Lifecycle hooks
  9. Permission & safety layer

1 · Outer iteration loop

The core architectural foundation is the while loop. Using the system prompt, the model decides — based on the data — which tools to call, then keeps composing tool after tool, execution after execution, until the job is done. This loop over tools is precisely what a human-configured state graph (LangGraph) is not.

2 · Context management & compression

The hard questions: what do you pull into context, what do you simplify or compress, and how do you represent large data compactly? Context management decides how the growing session trajectory is filtered, truncated, and structured before it reaches the model.

3 · Managing Agent Skills and tools

Every harness ships with a registry of built-in tools — read a file, edit a file, run a shell command, search code. These are the primitives. The harness knows what tools exist, what permission each requires, and how to dispatch a model's tool call to the right execution path.

Skills are a layer on top of tools. The harness discovers skill files from well-known directories, and the model invokes them by name. This is where organizational knowledge is encoded.

Tools are universal. Skills are specific to your team and your workflow.

4 · Managing subagents

At some point a task gets too big or too parallel for a single thread. Subagent management spawns child agents that work in isolation and report back. The key architectural decision is isolation: a subagent must not corrupt the parent's context or run tools the parent didn't authorize.

5 · Built-in, prepackaged skills

Every harness ships with baseline capabilities that work out of the box. The primitives are table stakes; the real differentiation comes from the quality of the higher-level skills the vendor has already figured out.

6 · Session persistence and recovery

A long-running session is stateful. If the process crashes you lose everything — unless the harness persists state to disk. Modern harnesses write incrementally as append-only JSONL: each message, each tool result, each compaction event gets a line. That means you can resume a session exactly where you left off.

7 · System prompt assembly & project context injection

The system prompt tells the model who it is, what it can do, and what it's working on. It is not a static string — it is assembled dynamically from multiple sources, stitched together within character/token budgets so it doesn't blow out the context window before the conversation even starts.

8 · Lifecycle hooks

Hooks are the extensibility seam. They let organizations inject custom logic before or after tool execution without forking the harness.

  • Pre-tool hook — fires before a tool runs; receives the tool name + input; can allow, deny, or modify the execution.
  • Post-tool hook — fires after; can inspect the result.

Hooks communicate via a structured protocol — JSON on stdin, exit codes for allow/deny — so they can be written in any language and enforce arbitrary policy: never run rm -rf, log all file writes to our audit system, require approval for any bash command touching production. This is how enterprises adopt harnesses: the vendor provides the architecture; hooks layer each org's safety, compliance, and workflow rules on top.

9 · Permission & safety layer

This is the component that separates a useful tool from a dangerous one. The permission layer defines what the agent may do and enforces it at every tool execution.

Take this with you

These nine components are not a wishlist — they are what every successful harness converged on independently. That convergence is the signal.

The deeper pattern: where the decisions live

Early harnesses hardcoded everything — fixed truncation limits, static permission rules, predetermined context budgets. The harnesses winning now push decisions to the model:

  • Not sure what to keep in context? Let the LLM decide.
  • Not sure which files matter? Let the LLM search.
  • Not sure when to compact? Let the LLM manage its own memory.

The harness provides the feedback scaffolding that makes intelligence work. Without a harness a model is open loop. With a harness the model can act on feedback, take action, and extend its own skills — which is why it works where the original ReAct loops did not.

A model on its own is a one-shot text generator: it answers and stops. A model inside a harness can read a file, edit it, run the tests, see them fail, read the error, fix the code, and run the tests again. That closed loop — act, observe, adjust — is what turns a language model into an agent.
