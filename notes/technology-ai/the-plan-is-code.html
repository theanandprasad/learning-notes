The Plan Is Code: Claude Code's Dynamic Workflows

By Aparna Dhinakaran (Arize AI) • Published Jun 10, 2026 • Saved Jun 9, 2026
Agent Orchestration Claude Code Multi-Agent Adversarial Verification

Dynamic Workflows is Claude Code's newest way of coordinating agents — and its key move is turning the plan into code: a deterministic script that spawns the agents, instead of a plan living inside one agent's conversation.

The shift

Every prior multi-agent setup kept the plan inside a conversation — a single agent held it in its context window, or an orchestrator gathered subagent results back into its own window. The bigger the job, the more the conversation drowned the orchestrator in coordination. Dynamic Workflows instead writes code that contains the agent orchestration loop. Results come back as values in the script's variables, not messages between agents, so you only ever see the final report.

The test run

They pointed it at a 30,000-file production codebase and told it to hunt for security bugs. The cost of a short list of issues:

Each task got a dynamically created set of roles: a finder that surfaces candidate issues, a verifier that tries to tear them down, a fixer to make changes, and a synthesizer that folds the survivors into one answer.

What it's made of

Three building blocks are enough to follow a run:

agent() — spawns one subagent and returns its text, or, with a schema, validated structured data.
parallel() — fans several agents out at once and waits for all of them before it returns. (A barrier.)
pipeline() — also fans out, but doesn't wait, so one item can be three stages along while another is still on its first. (A flow.)

Two limits sit underneath: at most 16 agents run at once (fewer on a weaker machine), the rest queued; and no single run may exceed 1,000 agents total — a backstop against a loop that never terminates. Their run used a pipeline across code areas (so a slow area's verification could proceed while others were still being scanned), with parallel calls nested inside to wait for all of a candidate's verifiers before judging it.

Schemas make output become input

What lets one agent's output become the next one's input is the schema. Each agent returned a typed object instead of prose, and the script read the fields directly. The verifiers were held to this shape:

VERDICT_SCHEMAconst VERDICT_SCHEMA = { type:'object', additionalProperties:false, required:['refuted','verdict','reason'], properties:{ refuted:{ type:'boolean' }, verdict:{ type:'string', enum:['real','not-real','uncertain'] }, reason:{ type:'string' }, }, }

A verifier can't hand back a vague paragraph — it must commit to a refuted boolean, a verdict from three allowed values, and a one-sentence reason, and the runtime makes it retry until the output fits. The script can then count how many verifiers returned refuted: false "without parsing a word of English."

The script is the orchestrator

The script is a scheduler; the agents are the worker processes it dispatches. The control plane is deterministic and runs no model of its own — every judgment call happens inside a worker, and once the run starts the agents are blind to one another. The spine of the sweep, lightly trimmed:

the orchestration spineconst perArea = await pipeline( AREAS, area => agent(finderPrompt(area), { phase:'Scan', schema: FINDINGS_SCHEMA, agentType:'Explore' }), (found, area) => Promise.all(found.findings.map(async f => { const verdicts = await parallel(LENSES.map(lens => () => agent(refuterPrompt(f, lens), { phase:'Verify', schema: VERDICT_SCHEMA, agentType:'Explore' }) )) const survivedVotes = verdicts.filter(v => v.refuted === false).length return { ...f, survived: survivedVotes >= 2 } })) )

One finder runs per code area; the script ranks candidates by severity, breaks ties by the finder's own confidence, and drops some to stay under a cap. Every survivor enters the verify stage, where three refuters examine it at once. A refuter is a verifier given an adversarial job — told to disprove the finding, not confirm it — so a candidate is kept only when at least two of the three fail to refute it.

The model picks the models

All 23 finders and 306 refuters ran on Haiku; only the final synthesizer ran on Opus. Left to itself, the model spent the cheap, fast model on the wide fan-out and saved the expensive one for the single closing judgment — quietly assuming a cheap model is good enough to find and break real security bugs. "Decisions like that fall to the model now… the work left to you sits at the two ends: framing the task going in and judging what comes back."

Most of the work is doubt

The run was lopsided toward skepticism: 23 finders fed 306 verifiers — ~13 disprovers for every finder. Most of the effort (and both the cost and the value) went into disproving findings, not producing them. The only thing that differed between a candidate's three refuters was a single line — the lens:

LENSES (the three angles)const LENSES = [ { name:'exploitability', instr:'Can attacker-controlled input actually reach and trigger this sink end-to-end? ...' }, { name:'trust-boundary', instr:'Does the tainted value truly cross a trust boundary? ...' }, { name:'mitigations', instr:'Is there existing parameterization, escaping, validation, sandboxing, or auth/RBAC enforcement that neutralizes this? ...' }, ]

A finding can look real and still die under any one lens — the code may be unreachable from outside, the input may be internal rather than attacker-controlled, or it may already be sanitized.

Three identical skeptics would have missed what three different angles caught.

The refuters weren't turned loose to nuke everything: the prompt held them to evidence, so each could refute a finding only with concrete, code-backed proof.

The funnel

So, do you even need it?

The honest verdict: reach for it when the job needs more breadth than a single context can hold, and you have some way to grade the output afterward (a test suite, a compiler, a second pass). When the work instead needs to loop until it's done, or to stop and wait for your judgment partway through, /goal or a simpler chain of staged runs fits better. Three things to hold onto:

You no longer build the coordination by hand. Told nothing about structure, the model produced the whole fan-out-and-verify machine and ran it well — moving the scarce skill off wiring agents together and onto framing the task and judging the result.
A cheap, wide sweep gives you suspects and a feeling of confidence. Scaling the run raised confidence in the findings much faster than it raised their accuracy — survivors still owe you a stronger, wider second pass before you act.
What lets the run scale is what makes it miss things. Cheap models and isolated, single-scope agents are what let hundreds run at once for a price you can stomach — and they're exactly why anything that depends on context spread across the codebase walks straight through the gate untouched.
Our role as engineers will increasingly become less about hand-crafting agent interactions and more about designing the high-level strategies — the schemas, the lenses, and the judgment criteria — that keep these systems precise, accountable, and aligned.— Aparna Dhinakaran

The deeper shift: when the coordination logic becomes a script you can save, iterate on, and version-control, the workflow turns from a black box into part of your codebase.
