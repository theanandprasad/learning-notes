Speculative Decoding: 2-3x Faster LLM Inference

By Avi Chawla (@_avichawla) • Published May 14, 2026 • Saved Jun 26, 2026
Inference Speculative Decoding LLM Optimization EAGLE/Medusa

A small model guesses several tokens ahead; the large model checks them all in a single pass. Correct guesses are free. The output is mathematically identical to running the large model alone — you just got there 2-3x faster.

Key idea

Google uses speculative decoding in AI Overviews to serve 1B+ Search users; Anthropic, Meta, and most major inference providers use it to cut latency at scale. The result is 2-3x more tokens/sec with mathematically identical outputs. The trick is borrowed from CPU architecture (speculative execution): a small draft model proposes several tokens ahead, the large model verifies them all in one forward pass, and the correct predictions — typically 60-80% of tokens — get accepted at the inference cost of the small model.

The problem: decoding is memory-bound

A 70B-parameter model in FP16 needs roughly 140 GB of GPU memory just to hold its weights. During the decode phase, generation is autoregressive: every single token requires a full forward pass through all 70B parameters, which means reading the entire set of weights out of memory.

On an H100 with 3.35 TB/s of memory bandwidth, just moving those weights takes about 42 ms to produce one token. The GPU's compute units sit mostly idle — the bottleneck is memory, not math. Worse, the token you spent all that bandwidth on is often a common, predictable word like "the", "is", or "of" that a 0.5B model could have gotten right.

How it works

The insight: verifying a guess is cheap, because a single forward pass can score many token positions at once. So let a tiny model do the sequential guessing, and let the big model do one batched check.

1 · Draft. A small model (e.g. Qwen2.5 0.5B, roughly 100x smaller than the target) generates 5 candidate tokens autoregressively. Each draft step costs only about 1-2% of one large forward pass.
2 · Verify in one pass. The large model (e.g. Qwen2.5 7B) processes all 5 candidates in a single forward pass, computing its own distribution at every position simultaneously.
3 · Walk & accept. Verification steps through the draft tokens in order. Suppose the large model agrees on tokens 1, 2, 3 but disagrees at token 4: tokens 1-3 are accepted, token 4 is replaced with the large model's own prediction (already computed in this pass), and token 5 is discarded.

Best case

All 5 drafted tokens accepted, plus 1 bonus token (the large model's free prediction at the next position) = 6 tokens from one large forward pass.

Worst case

All 5 rejected — but you still get 1 token, same as standard decoding. Rare, and only marginally more expensive than the baseline.

Why does this actually save time? Verification is structurally identical to prefill: many tokens scored together in parallel. That saturates the GPU's compute units instead of leaving them idle waiting on memory. And crucially, the output is mathematically identical to plain decoding because the accept/reject step is a proper rejection-sampling check — not a fuzzy "looks right enough" heuristic.

Correct predictions get accepted at the inference cost of the small model.

The variants

The core loop is always draft-then-verify. What changes across variants is how the draft is produced — and that determines acceptance rate, memory cost, and speedup.

VariantMechanismNotes / speedup
Separate draft model The baseline two-model setup: a small model from the same family drafts, the large model verifies. Needs a matched small model from the same family for best speedups; costs extra GPU memory for the drafter.
EAGLE Replaces the separate drafter with a lightweight head (<1B params) trained on top of the large model. Takes the hidden state from the second-to-last layer (rich semantic info) and predicts the next token without running the full model. A shortcut through just the EAGLE head — no separate model to host.
Medusa Multiple heads, each predicting a position ahead (head 2 → two ahead, head 3 → three ahead…), all run in parallel in one forward pass → predictions for the next ~5 positions at once. Heads are independent (head 2 doesn't see head 1's output), so candidates can be inconsistent → lower acceptance than EAGLE. Upside: no separate model or extra memory; heads are tiny (a few linear layers).
Self-speculative / early-exit Run only the first N layers and exit early with a prediction, K times → K draft tokens. The full model (all 80 layers) then verifies all K in one pass. Easy tokens get accepted from the early layers; harder ones rejected. More modest: 1.3-1.8x, since the drafter is the same large model run partially.
Diffusion-based drafting All the above drafters are still autoregressive (sequential). A diffusion drafter starts from a block of random noise tokens and refines them all in parallel → generates e.g. 16 tokens in one pass instead of 16 sequential steps. Drafting cost no longer scales with the number of proposed tokens. DFlash makes it practical: instead of a standalone diffusion model, it trains a small diffusion head conditioned on the target model's hidden states (like EAGLE), generating the whole draft block in parallel. Verification is unchanged.

Cross-tokenizer pairing (UAG)

Universal Assisted Generation (UAG) pairs any two models regardless of tokenizer, by matching sequences — for example a Qwen small model drafting for a Gemma large model. The convenience has a cost: same-tokenizer pairs give better speedups because the native path skips re-encoding.

Same tokenizer (native path)

1.5-3x speedup. Drafted tokens flow straight into the verifier — no re-encoding between models.

Cross tokenizer (UAG path)

1.5-1.9x speedup. Sequence matching bridges the two vocabularies, but adds re-encoding overhead.

A subtle gotcha

Even Instruct variants in the same family — e.g. Qwen2.5-7B-Instruct + Qwen2.5-0.5B-Instruct — can silently trigger the slower UAG path. Transformers compares tokenizer configs including chat templates, not just the vocabulary, so a chat-template mismatch is enough to kick you off the native route. Using the base models keeps you on the faster native path.

Production tradeoffs

The central tuning tension: a larger draft model improves the acceptance rate (its guesses agree with the big model more often) but increases drafting latency (each guess costs more). The sweet spot depends on the workload, and finding it is the main knob you tune in deployment.

Standard vs speculative decoding

Standard decoding

  • One full forward pass through the large model per token.
  • Memory-bound: ~42 ms per token just to move 140 GB of weights on an H100.
  • Compute units mostly idle, waiting on memory bandwidth.
  • Pays full price even for trivially predictable words.

Speculative decoding

  • Small model drafts several tokens; large model verifies them in one pass.
  • 60-80% of tokens accepted at the small model's cost.
  • Verification looks like prefill → compute units saturated.
  • 2-3x tokens/sec, mathematically identical output.
