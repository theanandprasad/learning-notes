Hierarchical Memory Management in Agent Harnesses

By Aparna Dhinakaran & team — Arize AI • Published Jan 29, 2026 • Saved May 31, 2026
AI Agents Agent Memory Unix / Filesystem Foundational

Why do the best agents use the file system and Unix commands as their memory layer? Because a fixed context plus tools to search and access files can be made to feel effectively infinite.

The thesis

The best agent memory systems aren't built top-down — they emerge bottom-up from composable primitives. The 50-year-old Unix philosophy (small, focused tools that do one thing well and compose infinitely) turns out to be exactly what LLMs need to make 200K tokens feel like 200 trillion.

This is the origin essay of the harness series — published before What is an Agent Harness and expanded by Context Management in Agent Harnesses. (Posted on X as "Agent Harness Architectures.")

The unreasonable effectiveness of Unix

What does a filesystem + search tools give an agent? The ability to make a fixed context feel infinite. Bash commands are powerful because they are composable (pipe one into the next) and they need no tool-definition JSON — the LLM already knows them. That is it.

Lessons from the history of memory

The 1980s challenge — build CPU memory hierarchies that make memory appear both infinite and fast — rhymes with today. The Commodore 64 had 64 KB and whole programs had to fit. The Intel 80486 popularized on-chip caches (frequently-accessed data appears fast) alongside virtual memory (memory appears infinite). For agents, the file system is memory, and agent tools provide the dynamic tradeoff between infinite storage and returning usable context.

Comparing Cursor, Claude Code, and Alyx

The "filesystem + Unix" approach shows up clearly in the success of Cursor and Claude Code, which operate over large data using simple Unix primitives. The same tools rapidly search, index, and locate information to greatly expand the available context window.

The evaluation: 10,000 files, find the Australians

Arize built an open eval: given 10,000 files of names and addresses where only one file holds Australian addresses (and nothing in the name reveals it), find and count them.

Input: "What postal zip codes do the Australian addresses have? Find those and count." Expected: Unique postal codes: 3 File 2000: 128 addresses (Sydney, NSW) File 3000: 134 addresses (Melbourne, VIC) File 4000: 738 addresses (Adelaide, Brisbane, Canberra, Darwin, Hobart, Perth) Total Australian addresses: 1,000

Dynamic self-correction & paging

The runs weren't flawless — and that's the interesting part. Claude recognized the data was too large for its context, backtracked, and found an approach that fit. Cursor used paging to step through results. Giving agents tools to right-size data is as powerful as giving them ways to bring it in.

Emerging technique: dynamic indexes

In databases, an index is a pre-computed structure answering "given this key, where is the data?" Unix commands like ls, find, grep have no pre-built index — they scan at runtime and produce output that functions like one: a map from query → locations, streamed through stdout.

The Unix philosophy treats command output as a universal interchange format (lines of plain text). Each command's output becomes the input — the "index" — for the next stage, letting the agent build ad-hoc query plans on the fly inside the context window.

Dynamic indexes with hierarchy in Alyx

Alyx goes further. It answers questions over huge volumes of trace data where a single row can be as large as the whole context window. The solution: a truncated preview table that acts as an index.

The future: can SQL replace Unix?

In observability there's no realistic way to bring all data local — it's too large. So: can you do with SQL what you do with Unix commands? The take: for simple retrieval, SQL can perform better; for complex operations (data augmentation, code-based tools), fetch the data to the filesystem first. Arize designed ADB to be fast for both real-time query APIs and filter-based file downloads, so agents choose the right tradeoff between remote querying and local execution.

Everything old is new again

Whether it's grep piping to sort, table previews pointing to full spans, or databases feeding file systems — the winning architecture is always a hierarchy of composable tools agents can chain together. Fifty years later, the lesson holds: make memory feel infinite by making it hierarchical, and make it hierarchical by making it composable.
