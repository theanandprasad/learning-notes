How We Use Skills (Lessons from Building Claude Code)

By Thariq (@trq212, Anthropic) • Published Mar 17, 2026 • Saved Jun 27, 2026
Claude Code Skills Agent Tooling Anthropic

Anthropic doesn't use a handful of skills in Claude Code — it uses hundreds, in active daily rotation. This is the engineer's-eye view of what they're for, how to write good ones, and how to share them without drowning the model in context.

The reframe

The common misconception is that skills are "just markdown files." That undersells them. A skill is a folder — it can carry scripts, assets, and data the agent can discover, explore, and manipulate. In Claude Code they also gain config options like dynamic hooks. The markdown is just the front door; the interesting part is everything behind it.

What a skill actually is

Once you stop thinking "instruction file" and start thinking "directory the agent can walk into," the design space opens up. The model can read the markdown to learn when and how to act, then reach for the scripts, reference data, and helpers sitting next to it.

The categories of skills they build

Across Anthropic's internal use, the hundreds of skills cluster into a handful of recurring shapes. The example names below are the payload — they show what a useful skill looks like in practice.

CategoryWhat it doesReal example skills
Coding & Design Make Claude better at your system, not coding in general. frontend-design — aligns output to your design system
Product Verification Describe how to test & verify code works, usually paired with an external tool. signup-flow-driver, checkout-verifier, tmux-cli-driver
Data & Query Query data with the right credentials, dashboard IDs, and canonical tables. funnel-query, cohort-compare, grafana
Business Process Collapse a repetitive workflow into one command; log results for consistency. standup-post, create-<ticket-system>-ticket
Debugging Stitch a single request's footprint across systems. log correlator — request ID → all matching logs
Infra Operations Routine maintenance/ops, some destructive — fenced with guardrails. <resource>-orphans, dependency-management, cost-investigation

Product Verification — worth an engineer-week

Verification skills are extremely useful… It can be worth having an engineer spend a week just making your verification skills excellent.

The reason verification pays off: it closes the loop between "the model wrote code" and "the code actually works." Two techniques carry most of the weight — have Claude record a video of its output to inspect, and enforce programmatic assertions on state at each step rather than trusting a final glance.

Data & Query — encode the tribal knowledge

The value here is not SQL skill — it's the institutional answer to "which events do I join, and what's the canonical key." A good data skill carries datasource UIDs, cluster names, and the table that holds the real user_id, so the model stops guessing.

funnel-query

Knows which events to join to see signup → activation → paid, and the table with the canonical user_id.

cohort-compare

Compares two cohorts' retention / conversion and flags statistically significant deltas.

Tips for making good skills

Don't state the obvious. Claude Code already knows a lot about your codebase and about coding in general. Restating it just burns context. Spend the words on what the model can't infer.
The description IS the trigger. "The field is not a summary — it's a description of when to trigger this skill." Write it as a routing signal, not a blurb.
Skills can hold memory. Append-only text log, JSON, or SQLite. standup-post keeps a standups.log, so the next run reads its own history and reports only the delta. Caveat: data in the skill dir may be wiped on upgrade — store in a stable folder. Today Claude Code provides ${CLAUDE_PLUGIN_DATA} per plugin.
Store scripts & generate code. "One of the most powerful tools you can give Claude is code." Hand it scripts/libraries and it spends turns on composition — deciding what to do — instead of rebuilding boilerplate.
Use on-demand hooks. A skill can ship hooks that activate only when that skill is called, so you get the behavior without it weighing on every session.

Sharing & distribution

For a small team across a few repos, just check skills into the repos — it works. But there's a hidden cost: every checked-in skill adds a little to the model's context. At scale that tax compounds, which is where a marketplace earns its keep.

How they manage it: no centralized gatekeeper team. Useful skills surface organically — upload to a sandbox folder in GitHub, point people to it in Slack, and once a skill gets traction the owner PRs it into the marketplace.

The catch

It's easy to create bad or redundant skills. Organic discovery only works if it's paired with curation — otherwise the marketplace fills with noise and the context tax returns through the back door.
