How to Create Loops with Claude

By MIKE (@mikenevermiss) • Published Jun 15, 2026 • Saved Jun 15, 2026
Claude Code Agentic Loops Automation Evaluator-Optimizer

Stop making prompts. Start designing loops. A prompt gets you one response; a loop gets you a system that keeps working after you close the laptop.

The shift in leverage

Boris Cherny (runs Claude Code at Anthropic) doesn't prompt Claude anymore — he has loops running that prompt Claude and figure out what to do. Peter Steinberger, same idea: don't prompt coding agents, design loops that prompt them. A loop is a recursive goal: you define a purpose, the agent iterates against it, and it runs until a real stopping condition is met. "The agent forgets everything between runs. The loop does not. That single fact is the entire architecture."

What a loop actually is — the six parts

Addy Osmani (a Google engineer who wrote the essay that named the practice) breaks a loop into six parts. Every working loop is some combination of these:

1 · Start with one trigger

Every loop starts with something that fires without you. Simplest version: a cron job that runs a Claude Code prompt on a schedule. Next: a hook — a script that runs automatically on an event like a commit or a file change. Pick one recurring task you do manually and turn the trigger into the first piece.

A complete, working loop on its own"Every morning at 8am, read yesterday's CI failures, open issues, and recent commits, and write findings to a markdown file."

Don't try to build the full six-part system on day one. "One automation that writes one state file is already more leverage than a hundred well-crafted prompts, because it runs without you."

2 · Give the loop a memory file

Create one markdown file — STATE.md or PROGRESS.md — where every iteration can read and write it. This is the loop's only memory. At the start of each run the agent reads it first; at the end it writes back what happened and what comes next. "This is the PROGRESS.md pattern, and it is the single most important file in any loop."

3 · Split the writer from the checker

The model that wrote the code is, in Osmani's words, "too nice grading its own homework." A single agent that writes and reviews its own work marks it done more often than it should. The fix is the evaluator-optimizer pattern (named in Anthropic's "Building Effective Agents" writeup):

4 · Isolate parallel work with worktrees

Once you're running more than one agent against the same codebase, isolation stops being optional. Give each agent its own working directory pointed at its own branch:

git worktree add ../agent-1-branch

5 · Set a hard stop condition

A loop without a real exit condition fails quietly. Geoffrey Huntley documented the "Ralph Wiggum loop": an agent meant to emit a completion signal only when finished emits it early, and the loop exits believing a half-done job is complete.

Real stop conditions

"The test suite passes." "The build succeeds." "The linked ticket moves to Done with a passing CI run." Checkable by something other than the agent's own claim.

Not a stop condition

"The agent says it's finished."

Set a maximum iteration count as a backstop regardless — 10 or 20 is a reasonable ceiling. If the loop hits the ceiling without meeting the real stop condition, it should halt and flag for review, not keep running.

6 · Wire in a human review checkpoint

Not every loop should run fully unattended from day one. Boris Cherny's autonomy ladder has four levels:

Runs that find something go to a triage inbox or flagged list. Runs that find nothing archive themselves silently — "you should never have to open a loop's output to confirm that nothing happened."

7 · Watch the token cost

"A single bad iteration is a wasted prompt. A single bad loop running unattended overnight is a bill." Each iteration is a full model call with the accumulated conversation history attached. Before letting any loop run unsupervised, estimate the worst case:

tokens/iteration × max iterations = worst-case cost / run worst-case / run × automation frequency = worst-case cost / day

Run it manually for 3–5 iterations first and measure tokens per iteration. And build a command allowlist for any loop that executes shell commands — restrict it to what the task needs (npm, git, ls, cat).

An agent with unrestricted shell access inside an unattended loop is the fastest way to turn a token-cost problem into a security problem.

8 · Build the second loop differently than the first

Your first loop should be small, single-purpose, and heavily supervised. Your second should connect to the first — this is where automations, skills, and memory compound instead of just running in parallel.
