Building a Context Engine for an Agent

By Cassidy Williams (DevRel, GitHub) · TLDR blog • Published Jun 12, 2026 • Saved Jul 7, 2026
Context Engine Agent Memory Retrieval Caching

An almost-working agent needed "a couple small things" — a little memory, and access to real data. That weekend project became a monster. This is the very funny, very real story of discovering that the thing you're actually building is called a context engine.

ℹ️ Sponsored/promotional piece. A Redis-sponsored essay on the TLDR blog; the back half is a soft pitch for Redis Iris (a Redis product). The problem framing is broadly useful and vendor-neutral; the recommended solution is the sponsor's — captured as one practitioner's take, not an endorsement.

The core idea

Giving an agent "a little memory and some data" is a trap: memory isn't one thing, data retrieval is its own infrastructure project, and caching sits on top. Put together, that layer has a name — a context engine: "the layer that gives an agent the right context, in the right shape, at the right time." It's most needed at runtime, while the agent is actually doing the work.

A "little memory" is… more than a little memory.

Memory is not one thing

Then the second wave: build vs buy. Store memory yourself? Use Postgres? A vector database? A memory framework? A model provider's built-in memory? A dedicated service? "Everything was a bit too reasonable, and scope creep loomed heavily with every option."

Data retrieval is a whole other infrastructure project

"At least passing in actual data should be straightforward — it's just one thing." (Editor's note in the piece: "I would like to throw the word 'just' into the sea.") Turns out data comes in flavors:

"Maybe the agent can just query Postgres?" opens its own can of worms — which tables can it access, which rows can this user see, what if the schema changes, what if it writes a bad query, or one that's technically correct but "wildly expensive," or the data is stale, or it finds the right record but misses the related ones… or the answer needs Postgres and Salesforce and Zendesk and "a PDF from 2014 named final_FINAL_v3.pdf." Raw data isn't usable; agents need the right data, shaped the right way, at the right time — and that "isn't just throwing search on it."

…and caching, of course

There are only two hard things in Computer Science: caching and naming things.
— Phil Karlton

Repeated questions, repeated retrieval, repeated LLM calls — all money and latency you pay again and again. So the last layer is caching (ideally semantic caching: reuse an answer when a new question means the same thing, not just when it's byte-identical).

All powers combined: the context engine

A context engine is the layer that gives an agent the right context, in the right shape, at the right time. At runtime it supplies: user state while answering · the latest data when deciding which tool to call · the relevant docs for planning · cached results to cut waiting · and memory so it doesn't repeat mistakes.

The "boring" infrastructure answer

The author's conclusion: she didn't want to build this herself anymore, and Redis "started to make a lot of sense… because it's boring in the best possible way." Developers already use it for fast, reliable infra that sits close to the runtime — hot data, caching, search, session state, queues, real-time workloads where latency matters. Agent context is very similar, and needs to sit close to where the user is.

Boring means fewer surprises, fewer custom glue layers… Now, I can build the agent. Redis can handle the context. No more drama.
