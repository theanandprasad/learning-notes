What We Learned Testing 7 Models Under the Same Agent Harness

By Nancy Chauhan — Arize AI • Published May 20, 2026 • Saved May 31, 2026
AI Agents Evals & Benchmarks Model Migration Harness

Model swaps look like configuration changes, but they behave more like product migrations. Change only the model — does the system still behave the way users expect?

The core lesson

A model swap is not safe just because the final answer still looks right. Under a fixed harness, correctness stayed in a tight band (79.6%–85.1%) — but operational behavior moved much more: latency, tool calls, retries, and timeout risk. Two models can both get the right answer while imposing very different cost, latency, and fragility on the system.

Part of the same harness series as What is an Agent Harness and Hermes Harness Architecture — here the harness is held fixed and the model is the variable.

Why this matters for agents

An agent is not just a model call. It is a model operating inside a harness: instructions, tools, schemas, state, retries, rate-limit handling, output expectations, and evals. From the outside a swap looks like changing a model name in a config file — but the product question is whether the whole system still behaves acceptably.

The experiment

A controlled GitHub-ops benchmark (acme-agent-evals) against a fictional "Acme SDK" repo seeded with issues, PRs, labels, milestones, and comments. Tasks were the kind a GitHub agent should handle: count open bugs, find issues with zero comments, identify PRs without reviews, compare milestone contents, audit labels and metadata.

Every model got the same task text, fixture repo, gh CLI access, pi.dev runner, GitHub CLI skill, and Arize evaluator suite. Only the model target changed.

Result 1 — correctness moved, but stayed in a tight band

The seven targets clustered between 79.6% and 85.1% correctness — a 5.5 percentage-point spread. Sonnet 4.6 was highest; GPT-5 lowest. Differences were measurable but contained enough to compare meaningfully.

Result 2 — correctness hid larger operational drift

The big differences showed up in how models reached the answer. Sonnet 4.6 averaged 9.5s / 1.45 tool calls; GPT-5 averaged 32.2s / 3.14 calls; Gemini 3 Flash used the most tools at 5.82 calls. A user may see the same final answer while the system absorbs very different latency, retry risk, and tool traffic.

This is the part of model drift that's easy to miss. It doesn't always show up as "the answer is wrong." Sometimes it shows up as "the product is slower, more expensive, more fragile, or more dependent on retries."

Result 3 — tool adherence saturated

Tool adherence hit 100% across every model — good news (models stayed on the allowed tool path), but also a warning: once a metric saturates, a binary "did it use tools?" check stops telling you much. The next question is tool discipline:

  • Did the model choose the smallest useful gh query?
  • Did it avoid unnecessary retries?
  • Did it handle pagination and empty states?
  • Did it transform JSON deterministically?
  • Did it use the tool when it should, instead of asking the user for help?

Tool adherence is a guardrail, not the whole eval.

Then we removed the harness

To test whether the harness was doing useful work, the same 19 tasks ran through a raw baseline: direct API calls with a small JSON protocol for requesting safe read-only gh commands — less scaffolding around how to ask for tools. A stress test, not a product comparison.

This benchmark behaved more reliably inside a fuller harness. One likely reason: the harness reduced the amount of protocol design each model had to rediscover on every task. It's consistent with the "stable harness hypothesis" — though it doesn't prove the harness caused all the stability.

What to do before you swap models

Treat model swaps like migrations, not vibes:

  1. Freeze the harness — same tools, prompts, fixture, dataset, and scoring.
  2. Run enough repeats to separate signal from noise.
  3. Score both final-answer quality and operational behavior.
  4. Inspect task-level failures instead of stopping at the average.
  5. Use the failures as the migration plan before changing production traffic.

The useful view is both

"Did the model get the job done — and what did the system have to do to make that happen?" If you measure only correctness, you may conclude the models are interchangeable. If you measure only latency, you may miss that a slower model is more complete. A model card tells you what changed in the model; your traces, evaluator results, and experiment comparisons tell you what changed in your product.
