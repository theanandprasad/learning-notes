A Claude Code Agent That Works in a Loop

By darkzodchi (@zodchiii) • Published Jun 23, 2026 • Saved Jun 26, 2026
Claude Code Agentic Loops Eval Gates Reusable Setup

Without a loop, an agent hands you broken work and leaves you to check it and re-run it yourself. You become the loop — and it's exhausting.

The key idea

Every self-fixing, self-optimizing, cost-cutting agent runs on the same pattern: do, check, repeat until done. Build that skeleton once and point it at anything — bugs, speed, cost. People write a separate "self-fixing agent," then a "self-optimizing agent," then another for cost, each from scratch. But they're all the same three moves: an agent does work, something measures whether it hit the goal, and a controller decides keep going or stop.

Get the skeleton right once and the specific job becomes a plug-in. Swap "run the tests" for "run the benchmark" and your bug-fixer becomes a speed-optimizer. The loop doesn't change, only the check does.

The three moves

Strip away the names and every looping agent is just three things wired in a circle. The agent does work. A check measures it against the goal. A controller reads the check and decides: go around again, or stop. That circle is the entire design.

File 1 — the reusable loop command

This is the skeleton: one slash command you write once and reuse forever. It lives at .claude/commands/loop.md. Notice it asks you what "done" means before it starts — that single question is the whole design.

.claude/commands/loop.md--- description: Work a task in a loop until a goal check passes argument-hint: <task> allowed-tools: Read, Write, Edit, Glob, Grep, Bash, Task model: sonnet --- Work this task in a loop: $ARGUMENTS 1. Establish the goal check. Ask me what "done" means if it isn't obvious: tests green, benchmark under X, no lint errors. 2. Do the work toward the goal. 3. Run the goal check. Record the result. 4. Met: stop and show me the result with proof. Not met: use the check's output to decide the next change, then go back to step 2. 5. Cap at 5 cycles. Announce the cycle number each round. The check is the truth, not your opinion. Never declare done without running it. Never weaken the check to pass it.

Where the design lives

The whole thing is in step 1: the loop asks what "done" means, then runs that as the truth every cycle. Everything else — the work, the decision — hangs off that one anchor.

File 2 — the pluggable check

The loop is generic because the check is swappable. Keep your checks as tiny shell scripts in .claude/checks/ and point the loop at whichever one matches the goal. A check is just something the agent runs and reads the output of.

.claude/checks/tests.sh#!/usr/bin/env bash npm test --silent 2>&1 | tail -3
.claude/checks/speed.sh#!/usr/bin/env bash node bench/run.mjs # prints a median ms number
.claude/checks/lint.sh#!/usr/bin/env bash npm run lint 2>&1 | tail -3

Same loop, three goals. /loop fix the failing auth test uses the test check; /loop make the search 2x faster points at the speed check. The skeleton never changes — you just tell it what "done" looks like.

File 3 — the stop rules

A loop needs to know when to quit — both when it has won and when it's spinning. These rules go in CLAUDE.md so every run inherits them.

CLAUDE.md## Loop stop rules Stop the loop when any of these is true: - The goal check passes. Stop, report success with the check output. - 5 cycles used. Stop, report what's left and what was tried. - The check result hasn't improved in two cycles. The agent is stuck. Stop and show me, don't burn cycle 4 and 5 guessing. - A cycle makes the check worse than the cycle before. Something is going backwards. Stop and surface it. Never report success without the check's output from the final cycle. Never edit the check itself to make it pass.

The rule that matters most

"Hasn't improved in two cycles" is what separates real progress from spinning. That's a human's cue to step in — not the agent's licence to keep grinding through cycles 4 and 5 on a hunch.

What you actually see

Two real runs. Same skeleton, same stop rules, two completely different jobs — one fixes tests, one chases speed. Watch the check drive each decision.

Common mistakes

Four ways people break the loop — each one quietly turns it back into manual work.

No goal check
A loop without a concrete "done" runs on the agent's opinion — and the agent always thinks it's done. The check is the loop. No check, no loop.
A check the agent can edit
It'll "pass" by weakening the test. The check must be something it runs, not something it can rewrite.
No cap
A stuck loop burns tokens until you happen to notice. Five cycles, then stop.
A new agent per goal
The whole point is one loop, many checks. Swap the check, keep the loop.

The 10-minute setup

Stand the whole thing up in one sitting. The trick at the end is to run it twice — once on each kind of goal — so you feel the skeleton stay still while the check changes.

3 min — Create the skeleton. Drop the loop command into .claude/commands/loop.md.
3 min — Write the checks. Add 2–3 scripts in .claude/checks/ for the goals you hit most.
2 min — Add the stop rules. Paste the loop stop rules into CLAUDE.md.
2 min — Run it twice. Run /loop on a real task, then again on a totally different one with a different check.
It didn't get smarter — you just stopped writing the same loop five times.
