Graph-Based Agent Memory

Neo Kim · System Design Newsletter · Jul 7 2026

When several agents share one project, each keeps its own memory — so what's missing is a shared memory that can read (follow connected facts), write (persist knowledge), and coordinate (many writers, no clobbering). Folders and vector DBs fail at coordination. The answer: a typed graph (nodes + edges + a shared schema) on object storage, with atomic writes via a manifest + compare-and-swap, "Git for AI memory" (branch → review → merge; agents never approve their own writes), Cedar policies, full audit/revert, and one query fusing graph + keyword + vector search via RRF. Case study: Omnigraph (sponsored).

Agentic Memory (for TanStack AI)

Tell a chatbot "I like sushi" and it forgets you the moment you start a new chat — because all it has is episodic (this-conversation) memory. Agentic memory adds durable, cross-chat facts via three operations: recall (fetch relevant facts before a turn, inject into context), retain (extract facts from the transcript after), and optional tools (remember ad hoc). Prompt Caching Is Everything (Lessons from Building Claude Code)

Thariq (@trq212, Anthropic) · Feb 20 2026

Long-running agents are made feasible by prompt caching, which works by prefix matching — anything that changes the prefix (including adding or removing a tool) invalidates the whole cache. So design around it: Claude Code keeps all tools loaded at all times and models plan mode as EnterPlanMode/ExitPlanMode tools rather than swapping the toolset; it defers tool loading instead of removing tools; and it monitors cache hit rate like uptime, declaring SEVs on cache breaks. Use messages, not system-prompt edits.
